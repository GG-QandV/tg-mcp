# Руководство разработчика (Developer Guide) — tg-mcpd v3.1 (Reactive Inbox)

Этот документ описывает архитектуру, механизмы реактивности и критические точки реализации реактивного почтового ящика в Telegram MCP-сервере (`tg-mcpd`).

---

## 1. Архитектура и сетевая топология

В версии **v3.1** реализована **Push-модель доставки сообщений** (активный пуш со стороны демона в API локального агента `opencode`).

```
Telegram (Чат)
  │
  │ MTProto Push (TCP persistent)
  ▼
tg-mcpd (daemon.py) ──> [InboxEngine (inbox.py)] ──> Запись на диск (inbox_store.py)
  │
  ▼
[InboxBridge (inbox_bridge.py)] ── (asyncio loop)
  │
  ├── Ждёт asyncio.Event (пришло реальное TG-сообщение)
  ├── Делает GET /session на порт нужного агента (Discovery)
  └── Делает POST /session/:id/prompt_async (Push)
        │
        ▼
   Агент opencode (порт 7771/7772/7773)
```

### Принцип изоляции агентов
Каждому топику в Telegram соответствует свой изолированный процесс ИИ-агента (`opencode`), работающий на собственном порту. Карта распределения задается в `.env` переменной `TG_TOPIC_MAP` в формате:
`"chat_id:topic_id:opencode_port,chat_id:topic_id:opencode_port"`

**Пример:**
```env
TG_TOPIC_MAP="-1003998609906:205:7771,-1003998609906:310:7772"
```

## 2. Разделение обязанностей и инбоксы

* **Telegram-клиент и демон (`tgmcpd`)** — строго **один процесс** на машину. Он авторизуется в Telegram, собирает сообщения со всей группы и раскладывает их по внутренним буферам.
* **Агенты (`opencode`)** — **несколько процессов** на разных портах. Демон пересылает сообщения в нужный порт на основе топика.
* **Персистентность**: Все сообщения записываются на диск в файлы `inbox_{chat_id}_{topic_id}.jsonl`. Сообщение удаляется с диска только после вызова `inbox_ack` (агентом или мостом).

## 3. Критические точки реализации (Узкие места / УМ)

При модификации кода почтового ящика разработчик обязан соблюдать следующие правила безопасности:

### [УМ-А] Безопасный Discovery (`_get_session_id`)
Метод `_get_session_id()` при отсутствии запущенного агента (пустой список сессий) должен возвращать `None` и не выбрасывать исключения.

### [УМ-Б] Сортировка сессий
При получении сессий от `opencode` необходимо брать сессию с наибольшим временем обновления, сортируя список:
```python
sessions.sort(key=lambda s: s.get("time", {}).get("updated", 0), reverse=True)
```

### [УМ-В] Порядок подтверждения (Ack)
Метод `InboxBridge` делает подтверждение (`ack()`) **только** после успешного выполнения пуша (получен HTTP-статус `204`). Если пуш провалился, сообщения остаются в хранилище.

### [УМ-Г] Безопасность при ошибках пуша
Если пуш не удался, сообщения не должны теряться. Они остаются на диске, и авто-пуш засыпает.

### [УМ-Д] Обработка 404 (Устаревший session_id)
Если агент перезапустился, его `session_id` меняется. При получении `404` в методе `_push()`, мост должен выполнить повторный Discovery (`_get_session_id()`) и повторить попытку отправки.

### [УМ-Е] Изоляция исключений в `daemon.py`
Метод `bridge.start()` и задачи `watch()` не должны ронять основной процесс демона при падении API агентов. Все исключения внутри `watch()` должны быть изолированы в `try/except`.

### [УМ-Ж] Отсутствие карты топиков
При пустом `TG_TOPIC_MAP` демон должен продолжать работу, записывая предупреждение `WARNING` в лог.

### [УМ-З] Безопасный цикл ретраев в `_push()`
При обработке `404` и поиске нового `session_id` в `_push()`:
- Обязательно сверять `new_id != current_session`. Если новый ID равен текущему или равен `None`, немедленно делать `return False` для избежания бесконечного цикла.
- Для корректного редисковеринга при `retry_max=1` (когда попытка с новой сессией не должна расходовать лимит сетевых попыток) использовать цикл `while` с контролируемым инкрементом счетчика попыток `attempt` вручную, а не классический `for`, чтобы `continue` не завершал цикл досрочно:
  ```python
  attempt = 0
  while attempt < self.retry_max:
      # ... r = post() ...
      if r.status_code == 404:
          new_id = await self._get_session_id(port)
          if new_id and new_id != current_session:
              current_session = new_id
              continue  # Повтор попытки без инкремента attempt
          return False
      # ...
      attempt += 1
  ```

### [УМ-И] Event-driven ожидание — поллинга нет
`inbox.wait()` **блокируется бесконечно** на `asyncio.Event.wait()`, просыпаясь исключительно от реального входящего сообщения из Telegram (вызов `handle()`). Никаких таймаутов, никакого поллинга.

Логика при запуске (restored messages):
- `restore_from_store()` загружает непрочитанные сообщения в RAM-буфер и выставляет `asyncio.Event`.
- Мост (`watch()`) делает **один** `push`-attempt.
- Агент недоступен → `clear_ram_buffer()` очищает RAM-буфер **и** сбрасывает Event.
- `wait()` засыпает до прихода **следующего нового** TG-сообщения. CPU = 0.
- Агент при старте сам вызывает `inbox_read` (→ `peek()`) для чтения сообщений с диска.

`peek()` всегда читает с диска (`store.read_all()`) если RAM-буфер пуст — гарантирует доступность всех накопленных сообщений для агента.

**Запрещено** добавлять в `wait()` таймаут или любой другой механизм поллинга.

## 4. Развертывание и тестирование

### Запуск тестов
Для корректного импорта модулей запускайте тесты строго с указанием пути к `src`:
```bash
PYTHONPATH=. python3 -m pytest tests/
```

### Безопасность Systemd-юнитов (Важно!)
При развертывании в пользовательском режиме (`systemd --user`) в юните [tgmcpd.service](file:///home/gg/.config/systemd/user/tgmcpd.service) **запрещено** использовать опцию `PrivateTmp=true`. 
Использование `PrivateTmp=true` изолирует сокет `/run/user/1000/tgmcpd.sock` внутри приватного пространства имен службы, делая его недоступным для клиентских процессов пользователя на хосте.
Без `PrivateTmp=true` сокет остается безопасным, так как права на директорию `/run/user/1000` (`0700`) и сам сокет (`0600`) разрешают доступ только владельцу и `root`.
